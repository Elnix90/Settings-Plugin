# DragonLauncher Settings Compiler Plugin

A Kotlin compiler plugin that automatically generates settings store property lists at compile time.

## Overview

This plugin eliminates manual maintenance of settings lists by scanning annotated classes and generating the complete list of properties automatically. It uses both the K2 Frontend (FIR) and backend (IR) compiler extensions to detect settings stores and inject the generated list.

## What It Does

The plugin processes classes annotated with `@SettingStore` and generates implementations for the abstract `ALL` property, which lists all settings in a store. Instead of manually maintaining a list like:

```kotlin
override val ALL: List<BaseSettingObject<*, *>> by lazy {
    listOf(
        this.setting1,
        this.setting2,
        this.setting3,
        // ... more settings
    )
}
```

You simply mark each setting with `@SettingKey`:

```kotlin
override val ALL: List<BaseSettingObject<*, *>> by lazy {
    TODO("Generated by plugin")
}

@SettingKey
val setting1 = ...

@SettingKey
val setting2 = ...

@SettingKey
val setting3 = ...
```

The plugin replaces the `TODO()` with the full list at compile time.

## Project Structure

This project contains three modules:

- The [`:compiler-plugin`](compiler-plugin/src) module contains the plugin implementation split into two compilation phases:
    - K2 Frontend (FIR) extension in the [`fir`](compiler-plugin/src/io/github/elnix90/settings/fir) package for declaration detection and generation
    - Backend (IR) extension in the [`ir`](compiler-plugin/src/io/github/elnix90/settings/ir) package for initializer code generation

- The [`:plugin-annotations`](plugin-annotations/src/main) module contains the `@SettingStore` and `@SettingKey` annotations used in user code.

- The [`:gradle-plugin`](gradle-plugin/src) module contains the Gradle plugin wrapper for easy integration into user projects.

## How It Works

The plugin operates in two compiler phases:

1. **K2 Frontend (FIR Phase)**: The `SettingStoreFirExtension` detects classes with the `@SettingStore` annotation and identifies existing `override val ALL` property stubs. It signals to the compiler that these properties need to be processed.

2. **Backend (IR Phase)**: The `SettingStoreTransformer` generates the actual implementation by collecting all `@SettingKey` properties and creating a `listOf(this.prop1, this.prop2, ...)` call to replace the stub.

For detailed information about how Kotlin compiler plugins work, see the [Kotlin compiler plugins documentation](https://kotlinlang.org/docs/custom-compiler-plugins.html).

## Extension Points

Extension point registration:

- K2 Frontend (FIR) extensions are registered in [`SettingPluginComponentRegistrar`](compiler-plugin/src/main/kotlin/io/github/elnix90/settings/SettingPluginComponentRegistrar.kt) via `FirExtensionRegistrarAdapter`.
- Backend (IR) extensions are also registered in `SettingPluginComponentRegistrar` via `IrGenerationExtension`.

The plugin follows the [FirDeclarationGenerationExtension](https://github.com/JetBrains/kotlin/blob/master/compiler/fir/providers/src/org/jetbrains/kotlin/fir/extensions/FirDeclarationGenerationExtension.kt) and [IrGenerationExtension](https://github.com/JetBrains/kotlin/blob/master/compiler/ir/backend.common/src/org/jetbrains/kotlin/backend/common/extensions/IrGenerationExtension.kt) extension points from the Kotlin compiler.
